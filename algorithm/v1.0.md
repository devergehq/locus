# The Locus Algorithm v1.0

Structured phased decomposition for agentic AI execution. Transitions from CURRENT STATE to IDEAL STATE using verifiable criteria.

## Effort Levels

| Tier | Budget | Criteria Range | When |
|------|--------|---------------|------|
| **Minimal** | <1min | 4-8 | Single, well-understood task |
| **Standard** | <5min | 8-16 | Normal request (DEFAULT) |
| **Extended** | <15min | 16-32 | Complex multi-file work |
| **Comprehensive** | <60min | 32-80 | Major architectural work, no time pressure |

## Ideal State Criteria (ISC)

Every criterion must be **atomic** — one verifiable end-state, 8-12 words, binary pass/fail.

**Splitting Test** — apply to every criterion before finalising:
1. **"And" test**: Contains "and"/"with"/"including" joining two verifiable things → split
2. **Independent failure test**: Can part A pass while part B fails? → separate criteria
3. **Scope word test**: "All"/"every"/"complete" → enumerate what "all" means
4. **Domain boundary test**: Crosses UI/API/data/logic boundaries → one criterion per boundary

**ISC Count Gate**: Cannot exit OBSERVE with fewer criteria than the effort tier minimum. If below, decompose until met.

## PRD as System of Record

The PRD (Product Requirements Document) at `{data}/memory/work/{slug}/PRD.md` is the single source of truth for any Algorithm execution.

**Frontmatter fields:** `task`, `slug`, `effort`, `phase`, `progress`, `started`, `updated`
**Body sections:** `## Context`, `## Criteria` (ISC checkboxes), `## Decisions`, `## Verification`

The AI writes all PRD content directly. Criteria use checkbox format: `- [ ] ISC-1: criterion text` → `- [x] ISC-1: criterion text` when satisfied.

## Phase Execution

### Phase 1: OBSERVE (1/7)

**Goal:** Understand the request deeply, define the ideal state, select capabilities.

**Context Gathering:**
- If a project index exists, query it for relevant context (2-4 queries from different angles)
- If past learnings exist, recall relevant ones for this task type
- Read relevant source files (targeted, not exploratory sweeps)
- If context gathering requires reading 3+ files, delegate to a sub-agent (if platform supports delegation)

**Reverse Engineering:**
```
REVERSE ENGINEERING:
  Explicit wants: [what they said they wanted, granular, one per line]
  Implied wants: [what they obviously want but didn't say]
  Explicit not-wanted: [what they said they don't want]
  Implied not-wanted: [what they obviously don't want]
  Gotchas: [common pitfalls for this type of task]
```

**Effort Level:** Select based on the reverse engineering output. Default to Standard.

**ISC Generation:** Write atomic criteria directly into the PRD. Apply the Splitting Test to every criterion. Count must meet the effort tier minimum.

**Capability Selection:** Review available skills and select those that will best accomplish the task within the time budget. Every selected skill creates a binding commitment to invoke it during BUILD or EXECUTE. Selecting a skill and not invoking it is worse than not selecting it.

```
CAPABILITIES SELECTED:
  [skill-name] — [phase it will be invoked] — [8-word reason]
```

**Checkpoint:** Write current state (ISC list, effort, capabilities, PRD path) to enable context recovery.

### Phase 2: THINK (2/7)

**Goal:** Pressure-test the plan before committing to execution.

```
RISKIEST ASSUMPTIONS: [2-8 assumptions that could be wrong]
PREMORTEM: [2-8 ways the current approach could fail]
PREREQUISITES: [things we need that might not exist]
```

Refine ISC criteria based on the premortem. Split any compound criteria. Add criteria for uncovered failure modes. Update the PRD.

### Phase 3: PLAN (3/7)

**Goal:** Validate prerequisites and establish the execution order.

- Validate all prerequisites identified in THINK
- Sequence the work (what depends on what)
- For Extended+: add a `### Plan` subsection to the PRD with technical approach and key decisions
- Re-evaluate capabilities — add any that the THINK phase revealed as necessary

### Phase 4: BUILD (4/7)

**Goal:** Prepare everything needed before execution.

- Invoke selected capabilities via the platform's skill/tool mechanism
- Any preparation work (scaffolding, research, design) happens here
- If the platform supports delegation, delegate investigative subtasks to sub-agents
- Document non-obvious decisions in the PRD's `## Decisions` section

### Phase 5: EXECUTE (5/7)

**Goal:** Perform the actual work.

- Execute the plan from Phase 3
- As each ISC criterion is satisfied, immediately mark it in the PRD (`- [ ]` → `- [x]`) and update the `progress` frontmatter field
- Do not batch criterion updates — mark them the moment they pass

### Phase 6: VERIFY (6/7)

**Goal:** Confirm every criterion is actually met, not just assumed.

For each ISC criterion:
1. Test that it's actually complete (not just "I think I did this")
2. Mark as verified in the PRD
3. Add evidence to the `## Verification` section

**Capability invocation check:** Confirm every selected capability was actually invoked. If any were selected but not invoked, flag as a failure.

### Phase 7: LEARN (7/7)

**Goal:** Extract insights to make future executions better.

```
LEARNING:
  What should I have done differently?
  What would a smarter approach have looked like?
  What capabilities should I have used that I didn't?
  What would I change about the algorithm for this type of task?
```

**Persist learnings** to `{data}/memory/learning/session/{YYYY-MM}/{YYYYMMDD}-{HHMMSS}_{slug}.md`:

```markdown
---
timestamp: {ISO 8601}
task: {task description}
project: {project path}
effort: {effort level}
---

# Learnings: {task description}

## Insights

{learning bullets — write every one, do not filter}

## Context

{1-2 sentences on what was built/changed}
```

Set PRD frontmatter `phase: complete`.

## Critical Rules

1. **Mandatory phase execution** — Every phase must be executed in order. No skipping.
2. **PRD is your responsibility** — If you don't write it, it doesn't get updated. No external system will do it for you.
3. **ISC Count Gate** — Cannot exit OBSERVE below the tier minimum. Decompose until met.
4. **Atomic criteria only** — Every criterion must pass the Splitting Test.
5. **No phantom capabilities** — Every selected capability must be invoked. Selection without invocation is dishonest.
6. **Context compaction** — At phase boundaries (Extended+), if context exceeds ~60% of working window, self-summarise. Preserve: ISC status, key results, next actions. Discard: verbose tool output, intermediate reasoning.
7. **Persist learnings** — The LEARN phase writing is mandatory. Learnings not written to disk are lost.

## Context Recovery

If after context compaction you lose track of state:
1. Read the most recent PRD from `{data}/memory/work/` — it has all state
2. PRD frontmatter has phase, progress, effort, task, slug
3. PRD body has criteria checkboxes, decisions, verification evidence

## Mode Selection

Before entering the Algorithm, classify the request:

- **Single file, single concept, one clear action, no investigation needed** → Handle directly (no Algorithm)
- **Everything else** → Enter the Algorithm

The Algorithm is for non-trivial work. Don't use it for renaming a variable or answering a question.

## Output Requirements (MANDATORY)

At every phase transition, produce visible structured output. Do NOT hide
reasoning in internal thinking blocks — the user must see your progress.

At minimum, each phase must visibly show:
- Phase name and number (e.g., "Phase 1: OBSERVE (1/7)")
- The key outputs defined for that phase (reverse engineering, ISC criteria, risks, plan, etc.)
- A brief progress summary before transitioning to the next phase

If the model supports hidden reasoning/thinking blocks, those may be used for
internal computation, but all Algorithm-required outputs MUST appear in the
visible response. Silent execution of Algorithm phases is a failure — the user
cannot verify compliance with phases they cannot see.
